# Vibe Coding 实战——从需求到产品的AI辅助开发过程

> **复盘笔记**：本文以中山大学3D标定场自动相机标定程序为案例，详细记录使用 AI 代理（GitHub Copilot Agent Mode）进行 Vibe Coding 的完整开发过程。内容涵盖需求表达、架构规划、迭代开发、问题诊断与解决等各环节，旨在为研究生提供 AI 辅助编程的实践参考。

---

## 1. 什么是 Vibe Coding

### 1.1 概念与定义

Vibe Coding 是一种新兴的软件开发范式：开发者用**自然语言**描述需求和意图，AI 代理负责生成、修改和调试代码。开发者的角色从"编写每一行代码"转变为"描述目标、审查结果、提供领域知识"。

传统编程教育要求先学语法、数据结构、框架，然后才能做项目。Vibe Coding 反转了这一路径：**先有问题，再找解法，代码只是解法的最后一个环节。**

### 1.2 适用场景：领域专家 + AI

本项目的开发者是摄影测量领域的研究者和教师——具备扎实的算法理论知识（相机模型、畸变校正、PnP求解），但并非专职软件工程师。这恰好是 Vibe Coding 最大的应用场景：

- **人**提供：问题定义、算法原理、物理约束、质量判断
- **AI**提供：代码实现、框架使用、调试排错、测试编写

两者互补，缺一不可。

### 1.3 工具链

本项目使用的开发环境：

| 工具 | 用途 |
|------|------|
| VS Code + GitHub Copilot (Agent Mode) | AI 辅助编程 |
| Python 3.10+ / OpenCV / NumPy / SciPy | 核心算法 |
| PySide6 | 图形界面 |
| pytest | 单元测试 |
| uv | 依赖管理 |

## 2. 项目启动：如何描述初始需求

### 2.1 第一条提示词

项目始于一条精心设计的提示词。以下是提交给 AI 代理的**原始需求描述**（摘自 `reference/prompt.md`）：

> **用户**：
> 
> 我是一个摄影测量的研究人员和教师，我希望为学校建立的相机标定场（3d test field）写一个自动标定程序。程序自动识别标定场中的目标点，并计算相机的内参和畸变参数。我想程序的运行流程如下：
> 1. 程序启动后，输入拍摄标定场的照片以及相机内参的初始估计值。
> 2. 程序自动检测照片中的标定点，并提取它们的像素坐标。
> 3. 程序提示用户输入若干个标定点的点号，程序根据标定场的结构自动匹配拍摄到的所有标定点的像素坐标与对应的世界坐标。
> 4. 程序使用匹配的像素坐标和世界坐标计算相机的内参和畸变参数，并输出结果。
> 
> 有关程序的编写，可以分两个阶段来实现：
> 1. 实现标定点的自动检测和匹配功能，确保程序能够正确识别标定点并与世界坐标进行匹配。
> 2. 实现相机内参和畸变参数的计算功能，并设计一个用户友好的界面来显示结果和提供交互功能。
> 
> 下面是给你提供的信息：
> 1. `reference\相机检校软件V1.0_使用说明.pdf` 是一个老旧的相机标定软件的使用说明文档……
> 2. `data\test_field\中山大学控制场-右手系.ocp` 是一个包含了标定场数据的文件……
> 3. 标定场宽约6 m，高约2.5 m，深约2.2 m。
> 4. `data\project\nikon-d800` 是示例数据……
> 5. `data\test_field\target_sample.jpg` 是非编码目标的图像。非编码目标为：白色底板（8×6 cm），中间有一个直径 3.8 cm的黑色圆，圆内有一个直径1.2 cm的黑白十字形标志。
> 
> **请只做架构设计，不要提供任何代码或代码片段。**

### 2.2 提示词设计的关键原则

回顾这条提示词，可以总结出以下设计原则：

**原则一：先说"我是谁"，建立领域上下文。**  
开头一句"我是一个摄影测量的研究人员和教师"，帮助 AI 理解用户的专业背景和技术水平，从而调整输出的专业程度和术语使用。

**原则二：明确运行流程，而非实现细节。**  
提示词描述了 8 个步骤的完整工作流程（从输入到输出），但没有规定用哪个 OpenCV 函数或哪种检测算法。这给了 AI 充分的实现空间，同时确保了最终产品满足业务需求。

**原则三：分阶段策略。**  
明确提出"分两个阶段实现"——先检测匹配，再标定界面。这避免了 AI 试图一次性生成整个项目的所有代码，降低了每一步的复杂度。

**原则四：提供具体数据，而非模糊描述。**  
提供了物理尺寸（"直径 3.8 cm的黑色圆"）、数据格式（.ocp、.int 文件）、示例数据（Nikon D800 照片）。AI 需要这些具体信息才能生成可运行的代码。

**原则五：先规划后编码。**  
最后一句"**请只做架构设计，不要提供任何代码或代码片段**"至关重要。它要求 AI 先输出项目结构和模块设计，供人审查和讨论，而不是直接生成可能方向错误的代码。

### 2.3 AI 的回应：架构设计

AI 在收到需求后，首先**主动研究了工作区中的数据文件**——读取 .ocp 文件以理解点号格式、读取 .int 文件以理解内参格式、查看目录结构。随后提出了分层架构方案：

```
src/calibration/
├── models/    # 数据类（无外部依赖）
├── io/        # 文件读写（依赖 models）
├── core/      # 核心算法（依赖 models, io）
└── gui/       # PySide6 界面（依赖 core, io, models）
```

依赖关系严格为 `models ← io ← core ← gui`，单向且无循环。这一架构设计在后续的整个开发过程中**未做修改**，验证了"先规划后编码"策略的有效性。

## 3. 为 AI 提供上下文：AGENTS.md 与文档驱动

### 3.1 AGENTS.md 的设计

`AGENTS.md` 是 AI 代理的**核心上下文文件**，每次新会话启动时自动加载。它的设计直接影响 AI 生成代码的质量和一致性。以下是其关键结构（摘录）：

```markdown
# AGENTS.md — 3D标定场自动相机标定程序

## 项目概述
- 为中山大学3D控制场（240个非编码目标点）开发自动相机标定程序。
- 技术栈：Python 3.10+, OpenCV, NumPy, SciPy, PySide6, Matplotlib

## 项目结构
src/calibration/
├── core/          # 核心算法（不依赖GUI）
│   ├── detector.py      # 目标检测
│   ├── matcher.py       # 自动匹配
│   ├── calibrator.py    # 标定计算
│   └── bundle_adjuster.py  # 光束法平差
├── io/            # 文件读写
├── models/        # 数据类
└── gui/           # PySide6 GUI

## 数据格式
### .ocp 文件（标定场世界坐标）
- 首行：点数（240）
- 后续行：点号\tX\tY\tZ\t标志位
- 注意：点号1609出现两次（数据重复），解析时取第一条

### .int 文件（相机内参初始值）
- cx/cy 为相对于图像中心的偏移量（需转换：cx_abs = width/2 + cx）

## 关键算法约束
### 畸变模型
Brown模型，4参数 (k1, k2, p1, p2)
## 编码规范
- 使用 type hints
- 数据类使用 @dataclass
- GUI 与算法严格分离：core/ 不得导入 gui/ 或 PySide6
```

### 3.2 AGENTS.md 的设计要点

**要点一：数据格式必须精确。**  
特别标注了 .int 文件中 cx/cy 为偏移量而非绝对坐标，以及 .ocp 文件中 1609 号点的重复问题。这些细节如果遗漏，AI 生成的解析代码就会出错，且错误难以发现。

**要点二：编码约束显式声明。**  
"core/ 不得导入 gui/ 或 PySide6"这一条约束，直接决定了代码的可测试性。没有它，AI 可能会在算法代码中引入 GUI 依赖。

**要点三：物理参数写入文档。**  
"白底板8×6cm，黑圆Φ3.8cm，十字Φ1.2cm"这些物理尺寸被写入 AGENTS.md，确保 AI 在任何会话中编写检测算法时都能获取这些信息。

### 3.3 设计文档体系

除 AGENTS.md 外，项目维护了 5 篇设计文档（`docs/` 目录下）：

| 文档 | 内容 | 作用 |
|------|------|------|
| `architecture.md` | 数据流图、模块接口定义 | AI 理解系统全貌 |
| `file_formats.md` | .ocp/.int 文件格式详细规范 | IO 模块编写参考 |
| `detection_algorithm.md` | 检测算法设计与演进 | 检测模块编写参考 |
| `matching_algorithm.md` | 匹配算法设计 | 匹配模块编写参考 |
| `calibration_model.md` | Brown 模型公式、BA 参数构成 | 标定模块编写参考 |

**核心理念：文档即上下文。** AI 代理的代码质量上限取决于你提供的上下文质量。模糊的描述会产生模糊的代码，精确的文档会产生精确的实现。

### 3.4 文档同步的纪律

在开发过程中，每次算法发生重要变更，都要求 AI **同步更新相关文档和 `progress_and_roadmap.md`**。例如：

> **用户**：请将目前的开发状态和问题更新到 `docs\progress_and_roadmap.md`，以便在新的 Session 中进行处理。

这种做法确保了跨会话的上下文连续性——当 AI 代理的会话超时或中断时，新会话可以通过阅读文档立即恢复上下文。

## 4. 分阶段迭代开发

### 4.1 Phase 1：核心算法

开发严格遵循架构的依赖关系，**自底向上**构建：

```
models → io → detector → matcher → calibrator → bundle_adjuster
```

每个模块的开发循环为：

1. **描述需求**：在对话中说明模块功能和接口要求
2. **AI 生成代码**：AI 根据 AGENTS.md 和设计文档生成实现
3. **AI 生成测试**：AI 同时编写 pytest 单元测试
4. **运行验证**：执行测试，检查通过情况
5. **修正迭代**：对失败的测试或不符合预期的行为进行修正

Phase 1 完成时，共有 31 个单元测试全部通过（`test_ocp_parser: 6, test_int_parser: 7, test_detector: 8, test_matcher: 6, test_calibrator: 4`），核心算法层稳固可靠。

### 4.2 Phase 2：图形界面

需求描述方式从算法规格转向**交互逻辑**：

- "程序启动后显示主窗口，左侧为图像查看器，右侧为匹配点列表和标定结果"
- "检测到的点用绿色圆圈标记，种子点用红色，匹配点用蓝色"
- "支持 --debug 模式，自动加载测试数据"

AI 生成了完整的 PySide6 界面，包括 `main_window.py`、`image_viewer.py`、`point_table.py`、`result_panel.py`、`detection_dialog.py`、`seed_input_dialog.py` 共 6 个模块。

由于 Phase 1 已确保 `core/` 不依赖 GUI，界面开发可以独立进行，且核心算法可以在命令行环境中独立测试和调试。

### 4.3 Phase 3：标定精度优化

这是最具挑战性的阶段。初始标定结果 RMS = 10.77px（参考值 0.28px），差距悬殊。优化过程涉及检测算法的三次迭代、匹配策略的多次调整、标定流程的重构。

详见第 5 节的案例分析。

## 5. 遇到问题如何解决——典型案例分析

以下通过五个真实案例，展示 Vibe Coding 中人与 AI 协作解决问题的模式。

### 5.1 案例一：cornerSubPix 为什么失败

**背景**：检测算法 v1 使用 OpenCV 的 `cv2.cornerSubPix` 进行亚像素定位，这是棋盘格标定中的标准做法。然而在本项目中，检测到的目标中心偏差达到 10-20 像素。

**人机协作过程**：

> **用户**：现在的目标检测算法的结果，大部分点的偏差看起来都比较大（>1 cm）。请后退一步，重新审视目标检测算法，是否需要采用别的类型的算法？

> **AI**：
> 
> **问题根因：`cornerSubPix` 用错了**
> 
> `cv2.cornerSubPix` 是为**棋盘格角点**设计的——它寻找的是灰度梯度形成"鞍点"（saddle point）的位置。而我们的目标中心是：
> - 一个 d=0.3cm 的**白色小圆**，不是角点
> - 周围是黑白扇区交替的圆形十字图案
> - 梯度场在这里不满足角点的数学条件
> 
> 结果就是 `cornerSubPix` 会漂移到附近某个"看起来像角点"的地方，导致 10-20 像素的偏差。

AI 随后提出了 6 个替代方案（A-F），从最简单的"直接使用椭圆中心"到最复杂的"最小二乘匹配 LSM"，并给出了每个方案的预期精度。

> **用户**：我觉得采用方案A + C比较合适，请先更新文档，再写代码。

**教训**：

1. **AI 也会犯"惯性思维"的错误。** `cornerSubPix` 在棋盘格标定中是经典做法，AI 在初始实现中复用了这一经验。但这里的目标几何特征完全不同，需要人基于领域知识指出问题。

2. **"退一步"是关键策略。** 用户没有要求"调整 cornerSubPix 的参数"，而是要求"重新审视算法"。这促使 AI 从问题的**根因**出发思考，而非在错误的方向上继续调优。

3. **要求"先更新文档再写代码"，** 确保算法变更被记录，后续会话可以继续。

### 5.2 案例二：13XX 大目标漏检——自适应阈值的内在缺陷

**背景**：检测算法 v2 采用了方案 A+C（亚像素椭圆拟合），整体精度提升至 RMS 0.48px，但前层（13XX 编号）约 10 个大目标全部漏检。

**人机协作过程**：

用户尝试调整自适应阈值的 `block_size` 参数（从 51 增大到 101），发现问题依然存在且速度明显变慢：

> **用户**：抱歉，尽管现在目标点检测的总数有了提升（169个），目前距离最近的一列目标（13XX 编号）仍然无法被检测，我将检测参数中的自适应块大小改成了101，也还是不行，检测的速度也慢了许多。请认真思考原因，**是否需要退一步，审视算法的设计**？

AI 经分析后给出了根因诊断（摘自 `docs/progress_and_roadmap.md`）：

> **根本原因：`cv2.adaptiveThreshold` 对大目标的内在缺陷**
> 
> 自适应阈值在 block_size × block_size 窗口内计算局部加权均值，当目标直径 d ≥ block_size 时：
> 1. **圆心区域**：窗口完全在暗区内 → 局部均值 ≈ 暗像素值 → 误判为背景
> 2. **圆边缘区域**：窗口跨越黑白边界 → 正确判定为前景
> 3. **结果**：实心圆变成碎裂的环形 → 圆度极低 (0.14-0.18) → 被过滤
> 
> | block_size | 与 d=82 的比值 | 效果 |
> |---|---|---|
> | 51 | 0.62× | 窗口远小于目标 → 严重中心掏空 |
> | 101 | 1.23× | 仅比目标大23%，仍有伪影 |
>
> **结论：自适应阈值是检测环节的错误工具选择。**

AI 提出多个替代方案，最终采用 `SimpleBlobDetector`（多阈值候选检测）替换自适应阈值。

**教训**：

1. **调参无效时，说明方法本身不适用。** 如果一个参数不管怎么调都不能解决问题，往往不是参数的问题，而是方法的选择错误。

2. **用户的角色是"发现异常并追问"。** 用户不需要知道自适应阈值的底层原理，只需要观察到"13XX 始终漏检"并向 AI 追问原因。

3. **数据日志是诊断的基础。** 运行日志中 `轮廓过滤(圆度): area=5186 circ=0.142 < 0.650` 这样的信息帮助 AI 定位了问题——面积 5186 与预期吻合（π×41²≈5281），但圆度仅 0.14。

### 5.3 案例三："按下葫芦浮起瓢"——Blob 合并优先级

**背景**：引入 SimpleBlobDetector 后，13XX 大目标检测成功，但整体标定精度从 v2 的 <1px 退化至约 4px。

**根因分析**：

v3 初版的候选合并逻辑为 `_merge_candidates(blob_candidates, contour_candidates)`，即 blob 候选优先。约 167 个目标同时被两条链路检出，使用了 blob 中心（精度 3-8px）替代了 v2 时期精度更高的轮廓中心（<2px）。

**修复**：

```python
# 修复前：blob优先（精度低）
all_candidates = _merge_candidates(blob_candidates, contour_candidates)

# 修复后：轮廓优先（精度高），blob仅补充未覆盖区域
all_candidates = _merge_candidates(contour_candidates, blob_candidates)
```

修改一行参数顺序，即恢复了 v2 的高精度，同时保留了 blob 对 13XX 的覆盖能力。

**教训**：

1. **修复一个问题时，必须验证是否引入了新问题。** 引入新组件时，测试不能只看"新功能是否工作"，还要回归测试"原有功能是否退化"。

2. **理解候选合并的语义很重要。** `_merge_candidates(primary, secondary)` 中 primary 的候选会覆盖 secondary 的近邻候选。参数顺序决定了优先级。

### 5.4 案例四：标定精度优化——从 RMS 10.77px 到 0.29px

**背景**：初始标定结果令人失望：$f_x = 6551$（参考值 6133，偏差 6.8%），RMS = 10.77px（参考值 0.28px）。光束法平差后结果无明显改善。

**人机协作过程**：

AI 编写了一个系统性的诊断脚本，对比了 5 种标定策略的效果：

```python
# 诊断脚本核心逻辑
cam1, _, _, rms1 = calibrate([matched], world_points, camera)
# 方案1: fx=6550.95, RMS=10.77 px

cam2, rvecs2, tvecs2, rms2, filtered2 = calibrate_iterative([matched], world_points, camera)
# 方案2: fx=6160.76, RMS=0.29 px, 内点=95

matched2 = match_points(detected, world_points, seed_pts, cam2, image=img) 
cam4, _, _, rms4, _ = calibrate_iterative([matched2], world_points, cam2)
# 方案4: fx=6133.99, RMS=0.44 px, 内点=176
```

运行结果揭示了关键洞察：

> **Root cause**: All calibration approaches give RMS≈10-12px and fx≈6450-6550. The issue is **matched point quality** — many of the 110 points have ~10-20px systematic coordinate errors due to PnP matching with inaccurate initial intrinsics (f=7175 vs true f=6133, 17% off).

根因不在标定算法或平差算法，而在**观测数据质量**。初始内参偏差 17%，导致 PnP 投影位置系统性偏移 → 部分匹配点被错误匹配 → 标定结果偏差。

**解决方案（两步走）**：

| 阶段 | $f_x$ | RMS | 匹配点 |
|------|--------|------|--------|
| 单次标定 | 6551 | 10.77 px | 110 |
| 迭代外点剔除 | 6161 | **0.29 px** | 95 |
| 用优化内参重新匹配 + 迭代标定 | **6134** | 0.44 px | **176** |
| 参考值 | 6133 | 0.28 px | — |

**教训**：

1. **"垃圾进，垃圾出"——永远先检查数据质量。** 当算法本身没有逻辑错误时，优先检查输入数据是否被污染。

2. **迭代策略的威力。** 单靠一次标定或一次平差都不够，关键突破来自"标定→剔除外点→重新匹配→再标定"的闭环迭代。

3. **先诊断后治疗。** AI 编写的诊断脚本同时对比了 5 种策略，快速定位了问题所在，而不是盲目修改单个模块。

### 5.5 案例五：23XX 假匹配——遮挡问题

**背景**：用户在界面上观察到部分 23XX 编号的匹配点位置明显错误——这些世界点实际上被前层目标遮挡，不应出现在这张照片中。

**人机协作过程**：

> **用户**：目前标定后重投影误差超过 10 px 的点多为 21XX、24XX、25XX、13XX。这些点都是绿圈的位置正确，然而 PnP 之后的蓝圈有很大的偏移（如图）。针对这一问题，能否设定一个阈值，强行将蓝圈和绿圈重合？

用户观察到了现象并提出了直觉性的解决方案（"强行重合"），AI 在实现 Snap-to-Detection 修正的同时，进一步分析了 23XX 假匹配的深层原因，并实现了基于相机坐标深度的遮挡检测算法。

**教训**：

1. **3D 场景中遮挡问题需要显式处理。** 标定场有三个深度层，前层目标会遮挡后层目标。2D 匹配算法如果不考虑 3D 遮挡关系，必然产生假匹配。

2. **用户的直觉方案可以作为出发点。** "强行将蓝圈和绿圈重合"并非最终方案，但它指向了正确的方向——匹配坐标与检测坐标的偏差需要被修正。

## 6. Vibe Coding 的关键技巧与模式总结

### 6.1 需求表达技巧

| 技巧 | 示例 |
|------|------|
| 提供物理参数而非模糊描述 | "直径 Φ3.8cm" 而非 "一个黑色圆" |
| 提供示例数据和预期输出 | 附带 .ocp 文件、测试照片、参考标定结果 |
| 描述"做什么"而非"怎么做" | "程序自动检测标定点" 而非 "用 cv2.findContours 检测" |
| 分阶段描述需求 | 先检测匹配，再标定界面 |
| 附带运行日志和截图 | 方便 AI 诊断问题 |

### 6.2 与 AI 协作的模式

**模式一：文档先行**  
先写 AGENTS.md 和设计文档，再开始编码。文档是 AI 的长期记忆；代码在每个会话中生成，文档在每个会话间传承。

**模式二：测试驱动**  
每个模块编写完成后立即运行测试。31 个单元测试是反复重构算法时的安全网——每次改动后运行 `pytest tests/ -v`，确保没有引入回归。

**模式三：渐进式开发**  
`models → io → core → gui`，每层稳定后再建上层。从不跳过底层直接写上层代码。

**模式四：问题诊断协作**  
人提供现象和日志，AI 分析原因和方案。人做出选择（"方案 A+C"），AI 负责实现。

**模式五：领域知识注入**  
算法原理和物理约束由人提供（"目标中心是白色圆，不是鞍点""Brown畸变模型用4参数"），实现由 AI 完成。AI 不了解你的硬件、目标、标定场，这些信息必须由人明确传达。

### 6.3 常见陷阱

| 陷阱 | 案例 | 应对 |
|------|------|------|
| 过度信任 AI 的初始方案 | cornerSubPix 用于非角点目标 | 运行后验证结果，不合理立即追问 |
| 在错误方向上持续调参 | 不断增大 block_size | 调参无效时考虑换方法 |
| 一次改动太多 | 同时引入 blob + 改 merge 逻辑 | 每次只改一个变量，逐步验证 |
| 忽略数据质量 | 初始内参偏差 17% 导致标定失败 | 先检查输入数据，再怀疑算法 |
| 缺乏跨会话的上下文传递 | 新会话不知道之前做了什么 | 更新 docs/progress_and_roadmap.md |

## 7. 项目成果与反思

### 7.1 最终成果

| 指标 | 结果 |
|------|------|
| 焦距精度 | $f_x = 6134$（参考 6133，误差 0.01%） |
| 重投影误差 | RMS = 0.44px（参考 0.28px） |

| 单元测试 | 33 个全部通过 |
| 代码模块 | 5 层 14 个文件 |
| 设计文档 | 5 篇 |

### 7.2 对教学的启示

**启示一：算法理解仍是核心竞争力。**  
AI 可以生成代码，但不能替你判断 cornerSubPix 是否适用、自适应阈值为什么失败、标定精度差的根因是数据还是算法。领域知识是人在 Vibe Coding 中不可替代的价值。

**启示二：文档驱动和测试驱动的习惯至关重要。**  
AGENTS.md 和设计文档确保了 AI 的上下文质量；31 个单元测试确保了反复迭代中的代码正确性。这两个"习惯"在传统编程中常被忽视，在 Vibe Coding 中反而更加重要。

**启示三："退一步审视"是突破瓶颈的关键策略。**  
当 AI 陷入局部优化（不断调参数）时，人需要介入，提出"是否需要退一步审视算法设计"这样的问题。这种**元认知**能力是人类在 AI 协作中最重要的角色。

**启示四：Vibe Coding 不是"完全不写代码"。**  
它是一种**新的分工方式**——人负责定义问题、提供约束、验证结果、做出判断；AI 负责代码实现、框架调用、测试编写。两者的协作深度决定了最终产品的质量。
